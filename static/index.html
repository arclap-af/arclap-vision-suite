Bulk picture filter

      
Point at a folder of 50k–15M images. The Suite indexes every one with YOLO, computes quality metrics, and lets you build complex filters with live previews. Designed for curating annotation candidates and producing client-ready subsets.

1 Pick the pictures + the model that will look at them

      
Either index a new folder, or re-open an existing scan to pick up where you left off.

    


    

      What's happening here?
      
You're telling the Suite which folder to scan and which YOLO model should look at every picture. Once you click Start indexing, the model runs on every image and writes to a tiny database what it saw (which classes, how confidently, plus brightness/sharpness/timestamp). That database is the foundation everything else queries — it's a one-time cost per folder.

      
    
        
  • Scan time: ~5–15 minutes per 50k pictures on a 3090, assuming the folder is on a local SSD. OneDrive cloud-only files are 20× slower — set "Always keep on this device" first.
    • 
        
  • Sample every Nth: set to 10 for a fast preview pass, then re-scan with 1 once you've validated your filter rules.
    • 
        
  • Model choice: custom-trained = your project's classes; stock = generic COCO (people, cars, trucks…).
    •

2 Scanning every picture

      
Streaming live from the YOLO scanner. ETA is shown in the log.

    


    

      What's happening here?
      
For every picture, the model runs once on the GPU, the detections are written to the scan database (class, confidence, count, bounding box), and three quality numbers are computed from a downsampled grayscale copy: brightness (mean pixel value), sharpness (Laplacian variance — high = crisp edges), and contrast (pixel std-dev). The filename is also parsed for a timestamp like YYYY-MM-DD_HH-MM-SS so date/hour filtering works later. You can leave this tab — when the scan finishes a notification fires.

4 What do you want to keep?

      
Build a filter from three angles: which objects must be in the frame, when the picture was taken, and what visual quality you'll accept. The match counter at the bottom updates live.

    


    

      How this works
      
A picture is kept only if it passes every section below. Inside each section, the values combine sensibly (e.g. "min" + "max" form a range; checked classes combine via the logic dropdown). Start loose, then tighten — the live counter tells you immediately how many pictures survive.

Refine tags with CLIP (optional, for max accuracy)

          
Heuristics get the easy ~85% right. CLIP zero-shot re-checks ambiguous frames using a vision-language model — usually +10-15 pp accuracy on borderline cases. ~3-5 min per 50k frames on your GPU. First run downloads ~890MB.

          

            
            
          

          
        


        

          
Import hand-labelled ground truth (optional)

          
Have a labels.json mapping filename → good/bad (or named tags)? Import it to override the heuristic for those exact frames. Manual labels win — heuristic and CLIP can't override them.

5 Sanity check — preview a sample

      
A random selection of 12 frames matching your rules. Click Non-matches to spot-check what's being excluded.

    


    

      What's happening here?
      
Before you commit to materialising thousands of files, look at actual matches. If the sample contains pictures you don't want, go back to Step 4 and tighten the rule (raise min confidence, restrict hours, drop a class). If the non-matches contain pictures you do want, your rule is too tight — relax it. Re-roll a few times to feel confident.

What CSI_V1 actually detects today

      
The shipped CSI_V1 model is trained on 16 classes with mAP@50 = 0.861 on the held-out validation split. The 40-class taxonomy in core/seed.py includes additional aspirational categories (PPE/helmet/Schutzhelm at id=32, several specialty machines) that are not yet in the trained weights. Training data for those is being collected via the Discovery queue and will land in CSI_V2.

      
Honest current state:

      
    
        
  • ✅ Trained & reliable (16): heavy machinery (excavator, loader, dump truck, concrete mixer, etc.), workers, common site vehicles
    • 
        
  • ⚠️ In taxonomy but NOT trained yet: PPE/helmet (cls 32), several specialty equipment classes
    • 
        
  • The PPE/helmet alert auto-silences when the loaded model has no helmet class — no false positives
    • 
      


      
Workflow (typical)

      
    
        
  1. 📹 Cameras — Add an RTSP URL, name + site, save
    2. 
        
  2. 🎛️ Mission Control — Pick the camera, see the live stream + charts + recent events + zones
    3. 
        
  3. 📐 Zones — Draw polygons on the camera view, set rules (allowed/forbidden classes, count limits)
    4. 
        
  4. 🎯 Events — Browse all detection crops + frames, filter by camera/class/zone, bulk-promote to training
    5. 
        
  5. 🔔 Alerts — Route important events to email/webhook with cooldown + per-class filtering
    6. 
        
  6. 👁️ Review queue — Discovery: low-confidence + unknown objects flagged for human triage; assign to existing classes or promote to a new class for retraining
    7. 
        
  7. 📼 Recordings — Auto-cleaned after 7 days unless promoted
    8. 
      


      
Operations

      
    
        
  • Watchdog auto-restarts cameras whose heartbeat goes stale (>60s). 5 crashes/hour → flagged red, manual reset required (avoids restart loops).
    • 
        
  • Disk sweep runs every 30 min: recordings > 7 days deleted, event crops > 30 days deleted (unless promoted), discovery crops capped at 5000/camera.
    • 
        
  • Detection events save 1 cropped JPG per detection + 1 annotated frame per detected-second per camera.
    • 
        
  • Engines: TensorRT FP16 (Deploy tab) and INT8 (calibrated on val/ split) for 2–4× inference speedup.
    •

_data/discovery_crops/*.jpg
_data/cameras.db          ← camera registry + sessions
_data/events.db           ← all detections
_data/discovery.db        ← review queue
_data/zones/<cam>.json    ← polygon rules
_data/alert_rules.json    ← routing rules
_data/alert_history.json  ← last 500 fires
